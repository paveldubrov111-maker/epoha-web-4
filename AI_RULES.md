## Твої обов'язки перед написанням коду

1. Визнач що саме треба зробити і де це живе в структурі проєкту
2. Перевір чи не існує вже подібного компонента, хука або функції
3. Перевір чи справді потрібен новий файл
4. Подумай чи можна зробити простіше

---

## Правила (суворі)

1. **Не ламай існуючу архітектуру** — компоненти, хуки, сервіси, типи
2. **Один компонент — одна задача** — якщо файл > 200 рядків, розбивай
3. **Не змішуй UI, логіку і доступ до даних** в одному місці
4. **Не дублюй код** — якщо щось повторюється 2+ рази, виноси
5. **Використовуй зрозумілі назви** — ніяких `data`, `item`, `thing`
6. **Не додавай нові бібліотеки** без вагомої причини
7. **Весь доступ до Supabase** — через `firebase.ts`-адаптер або `services/`
8. **TypeScript обов'язковий** — `any` лише у крайньому випадку
9. **Стилі** — лише Tailwind, системно і передбачувано
10. **Перед створенням нового** — перевір чи це вже не існує

## Технічний борг — знай про це

Ці файли вже великі — **не додавай до них новий код без розбивки**:

- `Budget.tsx` (~4700 рядків) — потребує рефакторингу
- `App.tsx` (~850 рядків) — занадто багато стану
- `IOAssistant.tsx` (~1380 рядків) — логіка змішана з UI

Нові фічі — завжди в окремі компоненти у `components/features/`.

## Структура проєкту

```
src/
  components/
    features/budget/       ← бюджет
    features/investments/  ← інвестиції (Analytics, Calculators, Portfolio)
    layout/                ← Header
    ui/                    ← базові UI-елементи
  services/                ← Supabase, API
  utils/                   ← форматування, маппінг
  constants/               ← config, colors, charts
  types.ts                 ← всі типи тут
  firebase.ts              ← Supabase-адаптер
  App.tsx                  ← кореневий компонент
```
