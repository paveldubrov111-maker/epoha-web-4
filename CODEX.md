# CODEX.md — epoha-web-4

> Цей документ є головним стандартом проєкту.
> Будь-який код, написаний людиною або ШІ, повинен відповідати цим правилам.

---

## 1. Місія проєкту

Простий, чистий, масштабований, швидкий у розробці та стабільний у роботі фінансовий додаток.

Увесь код має писатися так, щоб:
- його було легко читати
- його було легко змінювати
- його було легко масштабувати
- новий ШІ або розробник швидко розумів логіку
- жодне нове оновлення не створювало хаос

---

## 2. Стек проєкту

| Шар | Технологія |
|-----|-----------|
| UI | React + TypeScript |
| Стилі | Tailwind CSS |
| База даних | Supabase (PostgreSQL) |
| Аутентифікація | Supabase Auth (Google OAuth) |
| Анімації | motion/react |
| Іконки | lucide-react |
| Роутинг | немає (SPA з вкладками) |
| Стан | useState / useCallback / useMemo (локальний) |

**Правило стеку:** Не додавати нові бібліотеки без вагомої причини. Supabase — єдине джерело даних.

## 3. Архітектура

### Структура папок

```
src/
  components/
    features/
      budget/        ← компоненти розділу Бюджет
      investments/   ← компоненти розділу Інвестиції
        Analytics/
        Calculators/
        Portfolio/
    layout/          ← Header, Footer, навігація
    ui/              ← базові UI-елементи (кнопки, інпути, тощо)
  constants/         ← config.ts, colors.ts, charts.ts
  services/          ← робота з Supabase, зовнішніми API
  utils/             ← дрібні допоміжні функції
  types.ts           ← всі TypeScript типи
  translations.ts    ← мультимовність (uk/ru/en)
  firebase.ts        ← Supabase-адаптер (Firebase-сумісний шар)
  supabaseClient.ts  ← ініціалізація Supabase
  App.tsx            ← кореневий компонент
```

### Шари відповідальності

| Шар | Відповідальність |
|-----|-----------------|
| `components/ui` | Базові кнопки, інпути, модалки — лише відображення |
| `components/features` | Бізнес-логіка конкретних розділів |
| `components/layout` | Шаблон сторінки, хедер |
| `services/` | Запити до Supabase та зовнішніх API |
| `utils/` | Форматування, маппінг, дрібна логіка без стану |
| `constants/` | Конфіги, кольори, налаштування |
| `types.ts` | TypeScript-контракти — єдине джерело типів |

## 5. Головні принципи

### 5.1 Простота понад усе
Перевага завжди віддається простому рішенню, а не "розумному, але заплутаному".

### 5.2 Один компонент — одна відповідальність
Кожен компонент, хук, сервіс, функція мають одну чітку задачу.

### 5.3 Не змішувати UI і логіку
UI відповідає за відображення. Бізнес-логіка, обробка даних і запити не живуть всередині великих UI-компонентів.

### 5.4 Повторне використання
Якщо код або елемент повторюється більше ніж 2 рази — виносити в окремий компонент, хук або утиліту.

### 5.5 Передбачуваність
Одна й та сама задача в різних місцях проєкту вирішується однаково.

### 5.6 Мінімум магії
Не використовувати приховану логіку, надмірно складні абстракції або "хитрий код".

## 6. Правила для компонентів

- Максимум ~150–200 рядків. Якщо більше — розбивай.
- Один файл — один головний компонент.
- Назва пояснює призначення: `UserCard`, `PortfolioSummaryBlock`, `CreateProjectModal`.
- Props явні — компонент отримує лише те, що йому реально потрібно.
- Не зашивати дані, тексти і конфіги в UI.

---

## 7. Правила для сервісів

- Вся робота з Supabase — через `services/` або `firebase.ts`-адаптер.
- Не викликати Supabase напряму в 10 різних компонентах.
- Методи сервісу мають зрозумілі назви: `getUserById`, `createPortfolio`, `deleteTransaction`.

---

## 8. Правила для хуків

- Хуки відповідають за логіку, не за розмітку.
- Один хук — одна задача: `usePortfolioData`, `useBudgetFilters`.
- Не змішувати в одному хуку фетчинг, валідацію форм, роутинг і модалки.

---

## 9. Правила для TypeScript

- TypeScript обов'язковий. `any` — лише у крайньому випадку.
- Всі спільні типи — в `types.ts`.
- Локальні типи — поруч із компонентом.
- Для зовнішніх даних і props — типи завжди описані.

---

## 10. Правила для назв

| Що | Стиль |
|----|-------|
| Компоненти | `PascalCase` |
| Хуки | `camelCase`, починаються з `use` |
| Змінні та функції | `camelCase` |
| Глобальні константи | `UPPER_SNAKE_CASE` |
| Файли компонентів | `PascalCase.tsx` |
| Файли утиліт/сервісів | `camelCase.ts` |

**Заборонені назви:** `data`, `item`, `thing`, `stuff`, `test123`, `newDataFinal`.

---

## 11. Правила для стилів

- Єдиний підхід — Tailwind CSS.
- Не змішувати Tailwind з inline-стилями без потреби.
- Кнопки, відступи, радіуси — повторюються системно.
- Повторювані рішення стандартизувати в `components/ui`.

---

## 12. Правила для стану

- Локальний стан — локально (`useState`).
- Глобальний стан (`App.tsx`) — лише для: авторизації, теми, мови, курсів валют, портфелів.
- Не робити `App.tsx` звалищем нового стану.

---

## 13. Правила для помилок

- Кожна помилка має бути оброблена.
- Користувач бачить зрозумілий результат: успіх, помилку або стан завантаження.
- Технічні деталі — в консоль, не в alert (крім критичних помилок БД).

---

## 14. Правила для залежностей

- Перед додаванням нової бібліотеки: чи можна без неї? Чи вона не дублює наявне?
- Менше залежностей — краще.

---

## 15. Стандарт якості перед збереженням змін

Перед завершенням будь-якої задачі перевірити:
- [ ] Код читається легко
- [ ] Назви зрозумілі
- [ ] Немає дублювання
- [ ] Логіка не змішана з UI без потреби
- [ ] Структура не зламана
- [ ] Немає зайвих імпортів
- [ ] Немає мертвого коду
- [ ] Помилки оброблені
- [ ] Типи описані
- [ ] Рішення достатньо просте

---

## 16. Золоте правило

> Кожне нове рішення повинно робити проєкт чистішим, а не складнішим.

Якщо після додавання фічі важче читати код, важче знайти логіку або важче пояснити що відбувається — рішення погане, навіть якщо воно "працює".
